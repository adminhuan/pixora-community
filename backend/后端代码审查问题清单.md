## 一、严重问题（P0 - 必须立即修复，否则无法上线）

### S-01 数据库模型全部为空，使用单表JSON存储
- 状态：[ ] 未修复
- 文件：`src/models/*.model.ts`（全部20个文件）、`prisma/schema.prisma`
- 问题描述：
  - 所有模型文件内容都是 `export {};`，没有任何实际定义
  - Prisma Schema 只有一个 `DynamicRecord` 模型，所有数据存在 `payload Json` 字段里
  - 等于把 PostgreSQL 当成一个 JSON 文件用，完全丧失了关系型数据库的优势
  - 没有索引、没有约束、没有外键、没有枚举
- 修复要求：
  - 在 `prisma/schema.prisma` 中为每个模型创建独立的表
  - 定义所有 enum 类型（UserRole、UserStatus、PostStatus 等）
  - 定义所有 @relation 关联关系和外键
  - 添加所有 @@index 和 @@unique 约束
  - 运行 `prisma migrate dev` 生成迁移文件
  - 删除 `DynamicRecord` 模型和 `InMemoryModel` 类
  - 删除 `src/constants/inmemory-db.ts`
- 参考：`/后端/任务/02-数据库模型设计.md`

### S-02 编辑/删除操作没有权限校验（越权漏洞）
- 状态：[ ] 未修复
- 文件：`src/controllers/post.controller.ts`、`blog.controller.ts`、`snippet.controller.ts`、`project.controller.ts`、`question.controller.ts`、`answer.controller.ts`、`comment.controller.ts`
- 问题描述：
  - `update()` 和 `remove()` 方法没有检查 `req.user.id === resource.authorId`
  - 任何登录用户都能编辑或删除别人的内容
  - 这是最基本的安全漏洞
- 修复要求：
  - 所有 update/delete 操作必须先查询资源，校验当前用户是否为作者
  - 管理员（role=admin）可以跳过此检查
  - 校验失败返回 403 Forbidden
  - 示例：
    ```typescript
    const post = await postService.findById(id);
    if (post.authorId !== req.user.id && req.user.role !== 'admin') {
      throw new AppError('无权操作', 403);
    }
    ```

### S-03 JWT 默认密钥为弱字符串
- 状态：[ ] 未修复
- 文件：`src/config/index.ts`（第44-45行）
- 问题描述：
  ```typescript
  accessSecret: process.env.JWT_ACCESS_SECRET ?? 'replace_with_access_secret',
  refreshSecret: process.env.JWT_REFRESH_SECRET ?? 'replace_with_refresh_secret',
  ```
  - 如果生产环境忘记设置环境变量，将使用这些弱密钥
  - 攻击者可以轻易伪造 Token
- 修复要求：
  - 启动时检测 NODE_ENV=production 且密钥为默认值时，直接抛出错误阻止启动
  - 密钥最小长度校验（至少32字符）
  - 示例：
    ```typescript
    if (process.env.NODE_ENV === 'production') {
      if (!process.env.JWT_ACCESS_SECRET || process.env.JWT_ACCESS_SECRET.length < 32) {
        throw new Error('生产环境必须设置安全的 JWT_ACCESS_SECRET（至少32字符）');
      }
    }
    ```

### S-04 密码重置令牌用 Math.random() 生成
- 状态：[ ] 未修复
- 文件：`src/services/auth.service.ts`（第147行、第228行）
- 问题描述：
  ```typescript
  const resetToken = `reset_${Math.random().toString(36).slice(2, 12)}`;
  ```
  - `Math.random()` 不是加密安全的随机数生成器
  - 令牌只有10个字符，空间太小容易被暴力破解
  - 同样的问题出现在验证码生成处
- 修复要求：
  - 使用 `crypto.randomBytes(32).toString('hex')` 生成令牌
  - 令牌长度至少64个十六进制字符
  - 存入 Redis 时设置过期时间（如15分钟）

### S-05 验证码没有保存到 Redis，无法校验
- 状态：[ ] 未修复
- 文件：`src/services/auth.service.ts`（第219-231行）
- 问题描述：
  - `getCaptcha()` 生成了验证码文本和图片，但没有将验证码存到 Redis
  - 用户提交表单时后端无法验证验证码是否正确
  - 验证码功能完全失效
- 修复要求：
  - 生成 captchaId（UUID）
  - 将 captchaId -> captchaText 存入 Redis，设置5分钟过期
  - 返回给前端 captchaId 和 SVG 图片，不返回 text
  - 注册/登录时通过 captchaId 从 Redis 取出文本进行校验
  - 校验后立即删除（一次性使用）

### S-06 验证码和重置码明文返回给前端
- 状态：[ ] 未修复
- 文件：`src/services/auth.service.ts`（第205行）、`src/controllers/auth.controller.ts`（第62-64行）
- 问题描述：
  - 短信验证码在响应中直接返回 `code` 字段
  - 图形验证码在响应中返回 `text` 字段（答案）
  - 等于直接告诉攻击者验证码是什么
- 修复要求：
  - 响应中只返回 `{ sent: true, captchaId: "xxx" }`
  - 永远不返回验证码明文
  - 开发环境可以在日志中输出用于调试，但不能在接口响应中返回

## 二、中等问题（P1 - 上线前必须修复）

### M-01 路由前缀混乱，存在路径冲突
- 状态：[ ] 未修复
- 文件：`src/routes/index.ts`、`src/routes/user.routes.ts`、`src/routes/upload.routes.ts`、`src/routes/ranking.routes.ts`
- 问题描述：
  - 用户路由注册在根路径，`GET /:id` 被注册为 `/api/v1/:id`，与其他所有路由冲突
  - 上传路由是 `/image`、`/avatar` 而不是 `/upload/image`、`/upload/avatar`
  - 排行榜路由的 `/users/:id/*` 与用户模块路由重复
- 修复要求：
  - 用户路由加前缀：`apiRouter.use('/users', userRoutes);`
  - 上传路由加前缀：`apiRouter.use('/upload', uploadRoutes);`
  - 排行榜路由不要重复定义 `/users/:id/*`，改为 `/rankings/*` 下统一管理
  - 最终路径必须与前端任务文档完全一致

### M-02 点赞/收藏没有幂等性处理
- 状态：[ ] 未修复
- 文件：`src/controllers/post.controller.ts`（第45-58行），同类问题存在于 blog、project、snippet、comment 等控制器
- 问题描述：
  - `like()` 方法每次调用都执行 `increment('likeCount')`
  - 没有检查用户是否已经点赞
  - 用户可以重复点赞无限刷数
- 修复要求：
  - 先查询 Like 表是否已存在记录
  - 已存在则取消点赞（删除记录并 decrement）
  - 不存在则创建记录并 increment
  - 收藏同理
  - 使用事务保证原子性

### M-03 评论计数不同步
- 状态：[ ] 未修复
- 文件：`src/services/comment.service.ts`
- 问题描述：
  - 创建评论时没有更新目标内容（帖子/博客/项目等）的 `commentCount` 字段
  - 删除评论时也没有减少计数
  - 页面显示的评论数会与实际数量不一致
- 修复要求：
  - 创建评论后：`increment(targetType, targetId, 'commentCount')`
  - 删除评论后：`decrement(targetType, targetId, 'commentCount')`
  - 使用事务保证数据一致性

### M-04 搜索是全表扫描，性能极差
- 状态：[ ] 未修复
- 文件：`src/services/search.service.ts`（第22-27行）
- 问题描述：
  - 将所有帖子、博客、问题、项目、代码全部加载到内存
  - 用 `includes()` 做字符串匹配
  - 大小写敏感
  - 数据量超过1万条时会严重卡顿甚至 OOM
- 修复要求：
  - 使用 PostgreSQL 全文搜索（`@@` 操作符 + `to_tsvector`/`to_tsquery`）
  - 或者使用 Prisma 的 `contains` + `mode: 'insensitive'`
  - 结果缓存到 Redis（5分钟过期）
  - 搜索建议使用前缀匹配 + Redis 缓存

### M-05 关注/取关操作没有事务保证
- 状态：[ ] 未修复
- 文件：`src/services/follow.service.ts`（第23-26行）
- 问题描述：
  - 关注操作包含3步：创建 Follow 记录、更新 A 的 followingCount、更新 B 的 followersCount
  - 没有使用事务，中间一步失败会导致计数不一致
- 修复要求：
  - 使用 Prisma 的 `$transaction()` 包裹所有操作
  - 取消关注同理

### M-06 管理员更新用户没有字段白名单
- 状态：[ ] 未修复
- 文件：`src/controllers/admin/user.controller.ts`（第19行）
- 问题描述：
  - `update()` 直接将 `req.body` 透传给服务层
  - 可以修改任意字段，包括 password、role、points 等
  - 应该限制可修改的字段范围
- 修复要求：
  - 定义允许修改的字段白名单
  - 从 `req.body` 中只提取白名单内的字段
  - 示例：
    ```typescript
    const allowedFields = ['nickname', 'bio', 'role', 'status', 'mutedUntil', 'bannedUntil'];
    const updateData = pick(req.body, allowedFields);
    ```

### M-07 CORS 配置过于宽松
- 状态：[ ] 未修复
- 文件：`src/config/cors.ts`（第5-9行）
- 问题描述：
  - `!origin` 时直接放行（允许所有不带 origin 的请求）
  - `corsOrigins.length === 0` 时允许所有来源
  - 生产环境可能被跨站请求攻击
- 修复要求：
  - 生产环境必须配置明确的允许域名
  - `corsOrigins` 为空时应拒绝而不是全部允许
  - 开发环境可以放宽

### M-08 缺少 express-async-errors
- 状态：[ ] 未修复
- 文件：`src/app.ts`
- 问题描述：
  - 控制器方法都是 async，但没有导入 `express-async-errors`
  - 未捕获的 async 错误会导致 UnhandledPromiseRejection，服务器不返回响应
- 修复要求：
  - 在 `src/app.ts` 顶部添加 `import 'express-async-errors';`
  - 或者给每个路由处理器包裹 asyncHandler
